# Заметки по архитектуре

## Общая схема

У расширения три основных runtime-слоя:

1. **Content scripts**
   Работают в изолированном мире расширения на `vk.com`.
2. **Background service worker**
   Координирует скачивание и запросы на извлечение данных из страницы.
3. **Код извлечения в контексте страницы**
   Выполняется в `MAIN` world страницы, потому что React internals не всегда доступны из изолированного мира content script.

## Почему извлечение выполняется в MAIN world

Content scripts в Chromium работают в изолированном JavaScript-мире. DOM при этом общий, но page-owned свойства вроде `__reactFiber$...` и `__reactProps$...` часто недоступны напрямую.

Чтобы обойти это корректно:

- content script временно помечает нужный `.AttachVoice`-узел или карточку `[data-testid="clips-feed-item"]` токеном
- service worker вызывает `chrome.scripting.executeScript(...)`
- код извлечения запускается в `world: "MAIN"` и находит помеченный узел уже в контексте страницы

Так расширение сохраняет чистое разделение ответственности и при этом получает доступ к нужным React-данным страницы.

## Стратегия извлечения

Extractors используют защитный подход с несколькими fallback-путями:

- находят `.AttachVoice` или `[data-testid="clips-feed-item"]` по временному токену
- проверяют сам узел, его потомков и несколько ближайших предков
- ищут и `__reactFiber$...`, и `__reactProps$...`
- поднимаются по цепочке `fiber.return` на несколько уровней
- пытаются прочитать несколько известных путей к данным

Приоритет URL для голосовых:

1. `linkOgg`
2. `audioTrack.url`
3. `linkMp3`

Приоритет URL для клипов:

1. `files.mp4_480`
2. `files.mp4_360`
3. `files.mp4_240`
4. `files.mp4_144`
5. `files.hls`
6. `files.dash_sep`

## Ответственность файлов

### `src/content/content-script.js`

- общий shared runtime для content-модулей
- работа с `chrome.storage.local`
- bootstrap и синхронизация состояния ВКЛ/ВЫКЛ

### `src/content/voice-module.js`

- ищет голосовые сообщения VK
- вставляет кнопку скачивания
- отслеживает динамически появляющиеся сообщения
- формирует имя файла
- отправляет запросы на извлечение и скачивание в runtime расширения

### `src/content/clips-module.js`

- ищет карточки VK Clips
- вставляет кнопку скачивания поверх карточки
- отслеживает динамически подгружаемую ленту клипов
- выбирает лучший доступный URL и формирует имя файла

### `src/background/service-worker.js`

- инициализирует дефолтные настройки
- валидирует запросы на скачивание
- запускает извлечение данных в page context для голосовых и клипов
- вызывает `chrome.downloads.download`

### `src/popup/*`

- небольшой MV3 popup-интерфейс
- локальный переключатель ВКЛ/ВЫКЛ
- сохранение состояния через `chrome.storage.local`

## Ключевые проектные решения

### Без сборщика

Проект намеренно оставлен build-free:

- проще ручная загрузка в браузер
- проще код-ревью
- меньше лишней инфраструктуры для портфолио-проекта

### Защитный подход

Проект предпочитает безопасное поведение жёстким предположениям:

- невалидные URL отклоняются
- отсутствие React internals не валит расширение
- DOM-запросы выполняются best-effort
- настройки имеют разумные значения по умолчанию
- для VK Clips `hls` и `dash_sep` обрабатываются как явный best-effort fallback

### Минимум комментариев

Комментарии добавляются только там, где контекст неочевиден, в основном вокруг:

- разницы между isolated world и `MAIN` world
- извлечения данных из React internals
- fallback-логики
